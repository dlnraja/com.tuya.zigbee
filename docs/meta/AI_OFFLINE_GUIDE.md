## 🧠 3. Core Engine Classes & Methods Reference (Index)

To help offline AIs understand the behavior of each core file without internet access:

### 1. `TuyaZigbeeDevice` (`lib/tuya/TuyaZigbeeDevice.js`)
*   **Purpose**: The universal root base class from which all individual Tuya Zigbee drivers inherit.
*   **Key Methods**:
    *   `onNodeInit()`: Device initialization hook. Initiates capability detectors and setups the node listeners.
    *   `safeSetCapabilityValue(capability, value)`: Safely sets capability values. Automatically intercepts and corrects out-of-bounds or misaligned scales on the fly by calling `smartScaleValue(capability, value)`.
    *   `safeRegisterCapability(capability)`: Auto-scaffolds capability registration with basic listeners if native registration helpers are not defined.

### 2. `SmartBatteryManager` (`lib/managers/SmartBatteryManager.js`)
*   **Purpose**: Handles bidirectional synchronization and virtual state synthesis between battery percentages and low battery alarms to prevent empty placeholders on the Homey UI.
*   **Key Methods**:
    *   `setBattery(percentage)`: Sets the raw percentage. If the percentage falls below **15%**, it automatically triggers the `alarm_battery` capability to `true`.
    *   `setAlarmBattery(alarmActive)`: Sets the battery alarm. If the device does not report battery percentage, it synthesizes a virtual state (10% on alarm active, 100% on healthy state) to keep the UI clean.
    *   `_voltageToPercent(voltageMv, curve)`: Interpolates battery levels using non-linear discharge profiles (e.g., `CURVE_CR2032`, `CURVE_2xAA`).

### 3. `SmartEnergyManager` (`lib/managers/SmartEnergyManager.js`)
*   **Purpose**: Automatically detects and translates power metering parameters (voltage, current, power, energy).
*   **Key Methods**:
    *   `handleDP(dpId, value)`: Parses incoming Tuya DP frames and registers energy capabilities.
    *   `_smartDivisor(dpId, value, defaultDiv)`: Dynamically determines if an incoming value needs scaling based on realistic physiological bounds (e.g., voltage should be 100-260V, power should be < 5000W).

### 4. `ZclToHomeyMap` (`lib/adapters/ZclToHomeyMap.js`)
*   **Purpose**: Standard dictionary mapping ZCL attributes, Z2M expose properties, and ZHA clusters to standard Homey capability IDs.
*   **Key Helper**:
    *   `smartScaleValue(capability, value)`: A static, self-healing value corrector that analyzes raw numeric magnitude (e.g., distinguishing between temperature tenths vs hundredths or voltage millivolts vs volts) and applies precise scaling to protect the UI.

### 5. `Z2MConverterAdapter` (`lib/adapters/Z2MConverterAdapter.js`)
*   **Purpose**: Auto-scaffolds Homey drivers (`driver.compose.json` and `device.js`) from Zigbee2MQTT typescript definitions.
*   **Key Helpers**:
    *   `sanitizeDriverId(driverId)`: Removes duplicate/redundant words and hybrid suffixes (e.g., converting `switch_wall_switch` to `switch_wall`) to ensure clean canonical folders.
    *   `generateDeviceJS(config)`: Programmatically outputs fully functional capability registrations instead of blank stubs.

## ⚙️ 4. Tuya Private Protocol (EF00) Architecture Guide

Most Tuya Zigbee hardware does not use standard Zigbee Cluster Library (ZCL) commands for reporting state or receiving actions. Instead, they funnel all operations through a proprietary Tuya Cluster **`0xEF00`** (sometimes cluster ID `61184`).

### 📥 Payload Reception Flow (Read/Report)

1. **Zigbee RX**: The physical Zigbee transceiver of Homey receives an incoming frame on Cluster `0xEF00`.
2. **Device Routing**: The frame lands on `lib/tuya/TuyaSpecificClusterDevice.js`.
3. **Payload Parsing**: The raw byte payload is sent to `lib/tuya/TuyaEF00Parser.js`.
4. **DataPoint Extraction**: The parser splits the byte stream into one or more Tuya **DataPoints (DPs)**:
   * **DP ID** (1 byte): The identifier of the parameter (e.g., `1` for On/Off, `101` for temperature).
   * **DP Type** (1 byte): Encoding format (e.g., `0x01` Boolean, `0x02` Value/Integer, `0x04` Enum).
   * **DP Length** (2 bytes): Size of the value field.
   * **DP Value** (Variable): The raw value bytes.
5. **Manufacturer Profile Lookup**: `UniversalCapabilityDetector.js` calls `EnrichedDPMappings.getDPsForManufacturer(manufacturerId)` to fetch the correct scaling coefficients and capability matches.
6. **Capability Mapping**: Values are formatted and scaled (e.g., divided by 10 or 100 via `smartScaleValue`) and bound to standard Homey capabilities (e.g., `measure_temperature`).
7. **Homey Interface**: The capability is updated on the device object (`this.setCapabilityValue(capabilityId, value)`), reflecting instantly on the user's dashboard.

```
Incoming Zigbee Frame (Cluster 0xEF00)
    │
    ▼
TuyaSpecificClusterDevice.js (RX Hook)
    │
    ▼
TuyaEF00Parser.js (Splits stream into raw DP ID, DP Type, Raw Value Bytes)
    │
    ▼
EnrichedDPMappings.js (Queries manufacturer-specific DP-to-Capability profiles)
    │
    ▼
TuyaZigbeeDevice.js (safeSetCapabilityValue applies smartScaleValue self-healing)
    │
    ▼
Homey Device UI Updated (this.setCapabilityValue)
```

## 📐 5. Non-Negotiable Coding Rules (Zero-Defect Standards)

Any AI or developer generating code for this repository must strictly adhere to these rules. Failure to do so will trigger CI build blocks or Athom Store submittal rejections.

### 🔑 Rule F1: Combined Fingerprints (CRITICAL)
An device matching signature is only valid if and only if **both** the `manufacturerName` and the `productId` match simultaneously.
```javascript
// ✅ VALID
fingerprint: [
  { manufacturerName: '_TZ3000_xyz', productId: 'TS0002' }
]

// ❌ INVALID (Wildcards are strictly forbidden)
fingerprint: [
  { manufacturerName: '_TZE284_*', productId: 'TS0601' }
]
```

### 🔑 Rule F5: Case-Insensitivity Without `.toLowerCase()`
Do **NOT** write manual `.toLowerCase()` calls on raw `manufacturerName` or `productId` inside individual drivers. Doing so risks runtime exceptions if the value is missing or undefined. Instead, always route comparison checks through `CaseInsensitiveMatcher.js`.

### 🔑 Rule F6: Standard Manifest Settings Keys & Driver Folder Naming
Always ensure:
1.  The naming structure of Zigbee settings keys inside driver compositions adheres to:
    *   `zb_model_id` (NOT `zb_modelId`)
    *   `zb_manufacturer_name` (NOT `zb_manufacturerName`)
2.  Generated driver directories **must not have hybrid or mixed suffixes** (e.g., folders should be `switch_wall` and not `switch_wall_switch`).

### 🔑 Rule F8: Local-First Runtime Policy
No code executed on the Homey Pro runtime may attempt external cloud API requests. All translations, mappings, scaling operations, and battery/energy reconciliation must be evaluated fully offline.

## ⏰ 6. Automated Weekly Evolution & Scraping Engines

To continuously synchronize our device support fleet with external Zigbee ecosystems, the repository runs automated multi-source scrapers:

### Scraper Workflow (`scripts/automation/fetch-z2m-fingerprints.js`)
*   **Source 1**: Zigbee2MQTT's official typescript device definition databases (`tuya.ts` and others).
*   **Source 2**: Zigbee Home Automation (ZHA)'s official Python quirks (`ts0601_switch.py`, `ts0601_dimmer.py`, `ts0601_trv.py`, etc.).
*   **Behavior**: It downloads both files, uses robust, case-insensitive, quotation-flexible regex matching to extract signatures, deduplicates raw definitions, and merges them directly into the local `data/z2m-tuya-definitions.json` and `lib/tuya-dp-engine/fingerprints.json`.
*   **Cron Schedule**: Runs automatically every Monday morning to fetch external evolutions without human intervention.
